## Runtime perf (measure-first — DOD inline checklist)

Measured 2026-06-23 with `tools/bench.gd` (AddressBook ×1000 people = 156 KiB,
Godot 4.8.dev). Baseline throughput: build ~8.2 MB/s, decode ~4.3 MB/s
(slowest), pack ~18 MB/s, unpack ~15 MB/s. RT1/RT2 measured as regressions —
refuted, do not reland without new evidence. **RT4 measured as a confirmed ~10%
decode win and landed** — the per-dereference allocation, not the 3-hop or the
packed byte loop, is the decode bottleneck RT2 was looking for.

| id | pri | item | notes |
|---|---|---|---|
| RT1 | 🚫 | **Packed codec hot-loop allocs** | Refuted. Pre-sized output + index-cursor writes (`out[c] = b`) ran *slower* (pack 8.75 → 9.45 ms): GDScript `PackedByteArray[i] = b` is a per-element Variant set; the batched C++ `append_array` of a per-word `content` array beats it. The per-word alloc is not the bottleneck — manual byte loops are. Left as-is. |
| RT2 | 🚫 | **Reader `_buf()` 3-hop** | Refuted. Lazy-caching `msg.segments.segments[seg_id]` in the reader (bool guard + cached `PackedByteArray`) added a per-call branch for no win (decode 36.9 → 38.6 ms). The 3-hop is cheap (CoW ref share + fast property access); decode cost is allocation-bound (String materialization in `to_text`, `Array[Reader]` in list getters), not pointer-chase. Left as-is. |
| RT4 | ✅ | **`CapnTarget` pooled per Message** | Landed. `Message.follow()` and its callees allocated a fresh `CapnTarget` (RefCounted) on every pointer dereference — ~5 per person on the AddressBook payload, ~250k allocs across the decode bench. The target is a pure transient: `StructReader`/`ListReader.from_target` copy its fields out immediately and never retain it, and reads are single-threaded with no `await` between follows. So one reused `_scratch: CapnTarget` per `Message` (reset via `_null_target()` on the null/OOB/limit paths) is safe and drops the alloc. Correctness gotcha handled: `_target_from_pointer` clears the sticky `is_cap` flag at entry, since the struct/list arms never reset it and a prior capability target would otherwise leak `is_cap=true` into a reused struct/list target. Measured A/B/A interleaved (120 iters, min-of-3): decode **37.22 → 33.40 ms (~10% faster, 4.3 → 4.8 MB/s)**, repeatable across the sandwich; build/pack/unpack untouched (builder doesn't use `CapnTarget`). All 129 GUT tests pass. |
| RT3 | P3 | **`read_u64` naming** | `CapnWireWords.read_u64` returns a possibly-negative bit pattern; consider `read_u64_bits` to surface that. No callers today. |
| RT5 | ✅ | **Decode text-path alloc reduction** | Landed (bench-neutral — see the attribution below). Two surgical reductions on the text/data read path: (1) `StructReader`/`ListReader.get_text`/`get_data` no longer allocate a throwaway `ListReader` just to call `to_text()`/`to_data()` — text/data is always a non-composite `List(UInt8)`, so the byte span `(content_word, elem_count)` reads straight off the (RT4-pooled) `CapnTarget` via new `Message.text_from_target`/`data_from_target` (COMPOSITE arm falls back to the reader path, unreachable on well-formed input); (2) `CapnTextData.text_from` slices once (trim the trailing NUL off the span end before copying) instead of slicing the full span then re-slicing to drop the last byte. Removes ~5 `ListReader` + ~4 `PackedByteArray` allocs per AddressBook person. **Measured bench-neutral** (decode ~33.4 ms before/after, within run noise) because decode is String-construction-bound, not alloc-bound — kept as a strict allocation + call-site-clarity reduction, not claimed as a speedup. 133/133 GUT pass. |
| RT8 | 🚫 | **Eager segment-buffer ref on the reader (inline `_buf()`)** | Refuted *for this workload* — a genuine ~40% win on the pure scalar-read path, but a net wash-to-slight-loss on the (text-heavy) AddressBook bench, so not landed. The idea: cache the segment `PackedByteArray` as a `_seg` field on `StructReader`/`ListReader`, set once at fill time (`set_from_inline` / `ListReader.from_target`), so every primitive getter reads `_seg.decode_u32(...)` instead of `_buf().decode_u32(...)` — dropping a GDScript function-call frame + the `msg.segments.segments[seg_id]` 3-hop per scalar read. **Measured, isolated scalar path** (one reader reused, 200k×`get_u32`, best-of-60): **290 → 175 ns/read (~40% faster)**. **But** each reader now carries a `PackedByteArray` field (init + CoW ref-touch per construction), a cost paid per reader regardless of read count; break-even is ~2-3 scalar reads/reader. On AddressBook (1 scalar read + several text reads per person, and text bypasses `_seg` via `Message.text_from_target`), the per-reader cost isn't amortized → decode **30.8 → 31.3 ms (slightly worse)**. Distinct from refuted **RT2** (which cached lazily in `Message` behind a bool guard — added a branch; this caches eagerly on the reader, no guard). **Conditional, not unconditional like RT7**: would net positive on numeric-heavy schemas (network packets, game state, configs), negative on text-heavy. Land only with a scalar-heavy bench added to justify it honestly — the current AddressBook bench would misrepresent it as a pure regression. Reverted; not committed. The wholesale call-overhead kill remains GDExtension (RT6). |
| RT7 | ✅ | **Reader wrapper-collapse (StructReader + Reader double-alloc)** | Landed — **~8% decode** (clean A/B vs HEAD: 33.5 → 30.8 ms, 4.8 → 5.2 MB/s, repeatable). The generated typed `Reader` now `extends CapnReader.StructReader` instead of wrapping one via a `_r` field, so reading a nested struct / list element allocates ONE reader (the typed `Reader`) instead of a `StructReader` plus a wrapper around it. Runtime gains in-place fillers — `Message.fill_root(out)`, `StructReader.fill_struct(ptr_index, out)`, `ListReader.fill_struct(i, out)`, backed by `StructReader.set_from_target`/`set_from_inline` — that populate a caller-supplied reader rather than allocating; the static `from_*` and `get_struct`/`get_root` now route through them (one code path). Codegen: every Reader getter calls `self.get_*` (inherited), and struct/list-element getters + `read_<x>()` emit `var r := X.Reader.new(); <fill>(i, r)`. `X.Reader.wrap(sr)` is kept as a public/back-compat constructor (copies a StructReader's fields in — used by hand-written code + tests + the cross-file `common`/`shapes` pair, which stay old-style since `common` has no cgr fixture and `shapes` only resolves `CommonCapnp` in the combined request that built the goldens). Builders are untouched (not on the decode path). **Collision note:** a Reader getter now shares a namespace with inherited `StructReader` methods, so a *top-level* field named exactly `struct`/`text`/`list`/`u32`/… would emit `get_<stem>` shadowing the inherited method — a loud parse error in the generated file, not silent breakage. | RT6 | 🚫 | **String construction / interning / StringName** | Refuted, and the decode bottleneck re-attributed. **String construction is NOT the decode cost — it is ~1.5%.** Controlled A/B (patch `CapnTextData.text_from` to `return ""`, skipping slice + `get_string_from_utf8` + the `String` alloc, everything else on the path unchanged): decode **32.34 → 31.84 ms**, i.e. the slice + UTF-8 parse + `String` build for all 4000 text fields costs only **~0.5 ms = ~1.5%** (~125 ns/field). An earlier `bench_attrib.gd` row claimed "String materialization ~63%" — that was a **mis-attribution**: the comparison row dropped the whole text-field *access path* (pointer-follow + per-call Variant dispatch + `.length()`), not just the `String` build. **Engine-source basis (4.8):** `PackedByteArray.get_string_from_utf8` → `String::append_utf8` (`core/string/ustring.cpp:1751`) is single-pass — one `resize_uninitialized(p_len+1)` over-alloc to worst-case, decode loop, shrink — no length pre-pass; Godot `String` is UTF-32 (`char32_t`) so byte→String is a widen+alloc, but for short strings that's ~26 ns. **Refuted alternatives, all measured slower:** (a) `get_string_from_ascii` (`memcpy`→CharString + `append_latin1`) is *slower* than utf8 (extra copy) and only correct for pure-ASCII — a GDScript high-bit scan to gate it would cost more interpreter ops than it saves; (b) per-Message `String` intern cache: +5% unique / +4% even at ~99% hit (must build the `String` to key it, cache only adds a lookup); (c) `StringName` return: +57% (it is `String` construction *plus* a global-table intern, and semantically wrong — capnp Text is arbitrary UTF-8 payload, not an identifier). No offset/range variant of `get_string_from_utf8` exists, so the sub-range `slice` is forced, but slice+parse together are negligible. **Real decode cost = GDScript traversal machinery**: reader/wrapper allocation + per-element Variant dispatch (`get_list`/`get_struct`/`wrap`/`get_text`). Levers that remain: codegen wrapper-collapse (`StructReader` + generated `Reader` double-alloc, ~12% — full 32.5 vs raw-runtime 28.6 ms) and ultimately GDExtension to kill the per-element Variant dispatch (10–100×, the inline-checklist last resort). String construction is a dead end. |
